# 블록체인 기반 DAO 회비/예산 집행 PRD & SRS

## 1. 문서 개요

### 1.1 문서 목적

본 문서는 동아리, 팀, 소모임의 회비를 스마트컨트랙트 금고에 보관하고, 구성원 투표로 승인된 지출만 자동 집행하는 웹 서비스의 제품 요구사항(PRD)과 소프트웨어 요구사항(SRS)을 정의한다.

본 프로젝트의 핵심은 단순한 투표 앱이 아니라, 실제 가치가 있는 공동 자금을 투명하고 변경 불가능한 방식으로 관리하는 공동 금고 시스템이다.

### 1.2 문서 구성

본 문서는 하나의 Markdown 파일에 PRD와 SRS를 함께 포함한다.

### 1.3 프로젝트 한 줄 요약

동아리 회비를 스마트컨트랙트 금고에 보관하고, 구성원 투표로 승인된 지출만 자동 집행되도록 하는 블록체인 기반 공동 예산 관리 웹 서비스.

## 2. PRD

## 2.1 배경 및 문제 정의

### 기존 회비 관리 방식

일반적인 동아리나 팀의 회비 관리는 다음과 같은 방식으로 이루어진다.

- 회비를 총무 또는 회장 개인 계좌로 입금한다.
- 지출 내역은 엑셀, 카카오톡, 영수증 사진 등으로 공유한다.
- 구성원은 운영자의 정직성과 기록 관리를 신뢰해야 한다.
- 지출 승인 과정이 비공식적이거나 사후 공유에 그치는 경우가 많다.
- 기록이 수정되거나 누락되어도 구성원이 이를 검증하기 어렵다.

### 해결하려는 문제

- 공동 자금의 보관 주체가 개인에게 집중된다.
- 지출 승인 권한이 운영자에게 과도하게 몰릴 수 있다.
- 회비 입금, 지출 제안, 투표, 집행 기록이 분산되어 투명성이 낮다.
- 구성원이 지출 의사결정에 참여했는지 검증하기 어렵다.
- 예산 집행 조건이 충족되었는지 사람의 판단에 의존한다.

### 블록체인이 필요한 이유

- 회비 사용 내역을 누구나 검증할 수 있어야 한다.
- 운영자가 임의로 공동 자금을 송금할 수 없어야 한다.
- 지출 제안, 투표, 집행 결과가 변경 불가능한 기록으로 남아야 한다.
- 사전에 정한 승인 조건이 충족되면 스마트컨트랙트가 자동으로 송금해야 한다.
- 중앙 관리자 없이도 공동 금고의 상태를 신뢰할 수 있어야 한다.

## 2.2 제품 목표

### 핵심 목표

- 조직별 스마트컨트랙트 금고를 생성한다.
- 구성원이 회비를 입금하고 입금 내역을 확인할 수 있다.
- 구성원이 지출 제안을 생성하고 투표할 수 있다.
- 투표 조건이 충족된 지출만 자동 집행된다.
- 모든 예산 흐름을 웹에서 조회할 수 있다.
- 지출 목적에 대한 증빙 자료를 등록하고 구성원이 사후 확인할 수 있다.

### 비목표

MVP에서는 다음 기능을 제공하지 않는다.

- 법정화폐 직접 결제 및 은행 계좌 연동
- 실명 인증, KYC, 세무 처리
- 다중 체인 지원
- 복잡한 역할 기반 권한 체계
- 모바일 앱
- 오프체인 영수증 자동 검증
- DAO 토큰 발행 및 토큰 보유량 기반 투표

## 2.3 대상 사용자

### 주요 사용자

- 동아리 회장
- 동아리 총무
- 동아리 구성원
- 소모임 또는 스터디 멤버
- 팀 프로젝트 구성원
- 행사 준비팀

### 사용자 페르소나

#### 회장/총무

- 조직을 만들고 구성원 지갑 주소를 등록한다.
- 회비 현황과 지출 제안 상태를 관리한다.
- 단독으로 돈을 집행할 수 없다는 점을 구성원에게 설명해야 한다.

#### 구성원

- 본인의 회비 입금 여부를 확인한다.
- 지출 제안의 목적과 금액을 확인한다.
- 찬성 또는 반대 투표를 한다.
- 예산 집행 내역을 검증한다.

#### 발표/데모 사용자

- Sepolia 테스트넷 ETH로 회비 입금과 지출 집행 과정을 확인한다.
- 블록체인 적용 이유를 화면 흐름과 온체인 기록으로 이해한다.

## 2.4 핵심 사용자 시나리오

### 시나리오 1: DAO 생성

1. 회장이 지갑을 연결한다.
2. 조직명을 입력한다.
3. 필요한 경우 추가 구성원 지갑 주소를 등록한다.
4. 투표 승인 기준을 선택한다.
5. DAO 생성 버튼을 누른다.
6. Factory 컨트랙트가 해당 조직의 스마트컨트랙트 금고를 생성한다.
7. 생성된 DAO 대시보드로 이동한다.

### 시나리오 2: 회비 입금

1. 구성원이 지갑을 연결한다.
2. 본인이 속한 DAO를 선택한다.
3. 회비 입금 화면에서 금액을 입력한다.
4. 지갑에서 트랜잭션을 승인한다.
5. 입금 완료 후 현재 금고 잔액과 입금 내역이 갱신된다.

### 시나리오 3: 지출 제안 생성

1. 구성원이 지출 제안 생성 화면으로 이동한다.
2. 제목, 설명, 금액, 수신자 지갑 주소, 투표 마감일을 입력한다.
3. 지출 제안을 등록한다.
4. 제안은 투표 중 상태가 된다.
5. 구성원은 제안 상세 화면에서 투표할 수 있다.

### 시나리오 3-1: 제안 취소

1. 제안자가 본인이 생성한 제안 상세 화면에 접근한다.
2. 제안 상태가 투표 중이고 투표 마감 전인지 확인한다.
3. 제안자가 취소 사유를 입력하고 제안 취소를 요청한다.
4. 지갑에서 제안 취소 트랜잭션을 승인한다.
5. 스마트컨트랙트는 제안자, 제안 상태, 투표 마감 여부를 검증한다.
6. 조건을 만족하면 제안 상태가 취소됨으로 변경되고 취소 이벤트가 기록된다.
7. 취소된 제안은 투표, 집행, 증빙 등록을 허용하지 않는다.

### 시나리오 4: 지출 투표

1. 구성원이 제안 상세 화면에 접근한다.
2. 제안 내용과 현재 찬성률, 반대 수, 마감일을 확인한다.
3. 찬성 또는 반대를 선택한다.
4. 지갑에서 투표 트랜잭션을 승인한다.
5. 투표 결과가 화면에 반영된다.

### 시나리오 5: 자동 집행

1. 투표 마감 후 승인 조건 충족 시 제안 상태가 집행 가능으로 표시된다.
2. DAO 구성원이 실행 버튼을 클릭할 수 있다.
3. 스마트컨트랙트가 조건을 검증한다.
4. 조건을 만족하면 금고에서 수신자 주소로 금액을 송금한다.
5. 제안 상태가 집행 완료로 변경된다.
6. 집행 내역이 예산 내역에 기록된다.

### 시나리오 6: 지출 증빙 확인

1. 집행 완료 후 제안자가 영수증, 구매 내역, 사진, 거래명세서 등 증빙 자료를 등록한다.
2. 시스템은 증빙 자료 원문을 Cloudflare R2에 저장하고, 서비스 DB에는 저장 위치와 해시 등 메타데이터만 저장한다.
3. DAO 구성원은 제안 상세 화면에서 증빙 자료를 확인한다.
4. 구성원은 온체인 집행 내역과 오프체인 증빙 자료를 함께 보고 지출 목적의 타당성을 판단한다.
5. 필요 시 구성원은 별도 사후 검수 절차를 통해 구매 완료 여부를 확인한다.

### 시나리오 7: DAO 종료

1. 구성원이 DAO 종료 제안 생성 화면으로 이동한다.
2. 시스템은 진행 중 제안과 집행 가능 제안이 없는지 확인한다.
3. 조건을 만족하면 구성원이 종료 제안 사유를 입력하고 종료 제안을 생성한다.
4. 종료 제안에는 남은 잔액을 전체 구성원에게 균등 분배한다는 내용이 포함된다.
5. DAO 상태는 종료 투표 중으로 변경된다.
6. 종료 투표 중에는 입금, 신규 지출 제안 생성, 일반 제안 집행을 차단하고 종료 제안 투표만 허용한다.
7. 필요 시 종료 제안자는 투표 마감 전 종료 제안을 취소할 수 있다.
8. 투표 마감 후 일반 지출 제안과 동일한 승인 조건을 만족하면 DAO 구성원이 종료 실행 버튼을 클릭한다.
9. 종료 제안이 취소되거나 부결되면 DAO 상태는 활성으로 복귀한다.
10. 승인된 경우 스마트컨트랙트는 남은 잔액을 DAO 구성원에게 균등 분배한다.
11. 1인 반환액은 `DAO 잔액 / 구성원 수`로 계산하고, 나머지 wei는 DAO 생성자에게 반환한다.
12. 잔액 반환이 완료되면 DAO 상태가 종료로 변경된다.
13. 종료된 DAO는 신규 입금, 지출 제안, 투표를 허용하지 않는다.

- Sepolia 테스트넷 ETH 사용
- 실제 회비 관리 서비스처럼 설명하되, 데모 자산은 테스트넷 자산으로 제한
- DAO 생성부터 입금, 제안, 투표, 집행까지 하나의 흐름으로 발표

## 2.6 성공 기준

### 제품 성공 기준

- 사용자는 개인 계좌가 아닌 스마트컨트랙트 금고에 회비가 보관된다는 점을 이해할 수 있다.
- 사용자는 지출 제안이 투표 없이는 집행될 수 없다는 점을 확인할 수 있다.
- 사용자는 승인 조건 충족 시 자동 송금되는 흐름을 확인할 수 있다.
- 입금, 투표, 집행 기록이 화면에서 일관되게 조회된다.

### 기능 성공 기준

- DAO 생성 후 구성원 목록과 투표 기준이 확인된다.
- 구성원만 투표할 수 있다.
- 동일 구성원은 동일 제안에 중복 투표할 수 없다.
- 승인 조건을 만족하지 않은 제안은 집행되지 않는다.
- 잔액보다 큰 금액의 지출은 집행되지 않는다.
- 집행 완료된 제안은 다시 집행되지 않는다.
- 사용자는 온체인 기록이 지출 목적 자체를 자동 보장하지 않는다는 한계와, 증빙 자료를 통해 사후 확인해야 한다는 점을 이해할 수 있다.

## 2.7 주요 정책

### 투표 정책

MVP에서는 조직 생성 시 기본 투표 기준으로 다음 중 하나를 선택할 수 있다.

- 과반 찬성: 전체 구성원의 50% 초과 찬성
- 2/3 찬성: 전체 구성원의 66.7% 이상 찬성

지출 제안 생성 시에는 개별 제안에 한해 만장일치 조건을 체크박스로 선택할 수 있다. 만장일치가 선택된 제안은 조직의 기본 투표 기준보다 우선하여 전체 구성원 전원의 찬성이 필요하다. 여기서 전체 구성원은 투표 참여자가 아니라 DAO에 등록된 전체 구성원을 의미한다.

투표 승인 기준은 스마트컨트랙트에서 다음 수식으로 계산한다.

- 과반 찬성: `yesVotes * 2 > memberCount`
- 2/3 찬성: `yesVotes * 3 >= memberCount * 2`
- 만장일치: `yesVotes == memberCount`

투표 가능 시간은 `block.timestamp < deadline`으로 판단한다. `block.timestamp >= deadline`이 되면 투표는 마감된 것으로 보고, 제안 결과 확정과 집행 가능 여부 판단을 허용한다.

투표는 1인 1표이며, 구성원 지갑 주소 기준으로 판단한다.

MVP에서는 모든 지출에 투표를 필수로 적용한다. 소액 지출 예외, 사전 승인 예산 한도, 운영비 자동 집행은 MVP 범위에서 제외한다.

### 제안 상태

| 상태 | 설명 |
|---|---|
| 투표 중 | 제안이 생성되어 투표를 받고 있는 상태 |
| 취소됨 | 제안자가 투표 마감 전에 취소하여 더 이상 투표, 집행, 증빙 등록이 불가능한 상태 |
| 부결 | 투표 마감 후 승인 조건을 만족하지 못한 상태 |
| 집행 가능 | 투표 마감 후 승인 조건을 만족했고 아직 송금되지 않은 상태 |
| 집행 완료 | 스마트컨트랙트 송금이 완료된 상태 |
| 집행 실패 | 잔액 부족 등으로 송금이 실패한 상태 |

### DAO 상태

| 상태 | 설명 |
|---|---|
| 활성 | 입금, 제안 생성, 투표, 집행이 가능한 상태 |
| 종료 투표 중 | DAO 종료 제안이 생성되어 종료 투표만 허용되고 입금, 신규 지출 제안 생성, 일반 제안 집행이 차단된 상태 |
| 종료 | 잔액 반환이 완료되어 신규 입금, 제안 생성, 투표, 집행이 차단된 상태 |

### 권한 정책

| 사용자 | 가능 기능 |
|---|---|
| 비연결 사용자 | 서비스 소개, 지갑 연결 |
| 지갑 연결 사용자 | DAO 생성 |
| DAO 구성원 | 회비 입금, 지출 제안 생성, 본인 제안 취소, 투표, 집행 실행, 종료 제안 생성 |
| DAO 운영자 | DAO 생성자로 영구 기록되며, 일반 구성원과 동일한 자금/투표/제안 권한을 가짐 |
| 비구성원 | DAO 생성, 컨트랙트 주소 또는 트랜잭션 링크를 통한 외부 검증 |

MVP에서는 DAO 생성자를 영구 운영자로 기록한다. 운영자 변경, 운영자 추가, 운영자 제거는 제공하지 않는다.

운영자는 구성원의 투표권을 변경할 수 없고, 승인 조건을 우회해 자금을 집행할 수 없으며, 투표 없이 금고 자금을 송금할 수 없다.

MVP에서는 DAO 생성 이후 구성원 추가/삭제 기능을 제공하지 않는다. 구성원 변경은 향후 확장 기능으로 검토한다.

지갑 연결 후 DAO 목록에는 연결된 지갑이 구성원으로 등록된 DAO만 표시한다. 비구성원에게는 다른 DAO 목록, 제안 상세, 증빙, 예산 상세를 웹 서비스 화면에서 제공하지 않는다. 외부 공개는 발표 또는 검증 목적의 컨트랙트 주소, 트랜잭션 링크 수준으로 제한한다.

### 지출 목적 검증 정책

블록체인이 직접 보장하는 것은 "누가, 어떤 제안에 투표했고, 승인된 금액이 어떤 수신자 주소로 송금되었는가"이다. 반면 송금 이후 실제 현실 세계에서 해당 금액이 제안 제목이나 설명대로 사용되었는지는 블록체인만으로 자동 검증할 수 없다.

예를 들어 "설비 구입"이라는 제안이 승인되어 특정 지갑으로 송금되었더라도, 수신자가 실제 설비를 구매했는지는 온체인 기록만으로 확정할 수 없다. 따라서 본 서비스의 신뢰 범위는 "승인 없는 자금 집행 방지"와 "승인 및 송금 과정의 투명성 보장"이며, "현실 지출 목적의 진실성"은 증빙 자료와 구성원 확인 절차로 보완한다.

MVP에서는 다음 방식으로 지출 목적 검증을 보완한다.

- 지출 제안 생성 시 제목, 설명, 금액, 수신자 주소를 명확히 입력한다.
- 가능하면 지출 담당자 개인 지갑이 아니라 실제 판매자 또는 거래처 지갑으로 직접 송금한다.
- 집행 완료 후 영수증, 구매 내역, 사진, 거래명세서 등 증빙 자료를 등록한다.
- 증빙 자료는 집행 완료된 지출 제안에만 등록할 수 있고, 온체인에 직접 저장하지 않고 Cloudflare R2에 저장한다.
- 제안 제목, 설명, 금액, 수신자 주소 등 제안 상세 데이터는 정해진 canonical JSON 표준으로 정규화한 뒤 `keccak256` 해시를 온체인에 저장한다.
- 증빙 자료는 파일별 `SHA-256` 해시를 별도로 생성해 DAO 금고 컨트랙트에 기록한다.
- 증빙 자료는 삭제/수정/교체를 허용하지 않고, 필요한 경우 추가 증빙만 등록한다.
- 증빙 등록에 필요한 온체인 해시 기록 트랜잭션의 가스비는 증빙 등록자가 부담한다.
- DAO 구성원은 제안 상세 화면에서 집행 기록과 증빙 자료를 함께 확인한다.
- 필요 시 큰 금액의 지출은 계약금, 중도금, 잔금처럼 단계별 제안으로 나누어 승인한다.

향후 확장 기능으로는 집행 후 구매 완료 여부를 별도 사후 검수 투표로 확인하는 방식을 고려할 수 있다.

## 3. SRS

## 3.1 시스템 개요

본 시스템은 Cloudflare Pages 기반 웹 프론트엔드, Cloudflare Workers 또는 Pages Functions 기반 API, 블록체인 지갑, Factory 컨트랙트, 조직별 DAO 금고 컨트랙트, 블록체인 네트워크, 서비스 DB, Cloudflare R2 파일 스토리지로 구성된다.

### 시스템 구성 요소

| 구성 요소 | 역할 |
|---|---|
| 웹 애플리케이션 | 사용자 인터페이스 제공. MVP 배포 대상은 Cloudflare Pages |
| API 계층 | Cloudflare Workers 또는 Pages Functions에서 실행되는 HTTP API, 오프체인 데이터 저장, 권한 검증, 해시 생성, 파일 업로드 흐름 처리 |
| 지갑 | 사용자 인증 및 트랜잭션 서명 |
| Factory 컨트랙트 | DAO 생성 요청을 받아 조직별 DAO 금고 컨트랙트를 배포하고 생성된 DAO 주소를 기록 |
| DAO 금고 컨트랙트 | 조직별 금고, 구성원, 제안, 투표, 집행 로직 관리 |
| 블록체인 네트워크 | 트랜잭션 처리 및 상태 저장 |
| 블록체인 RPC | 웹 애플리케이션과 네트워크 연결 |
| 서비스 DB | 온체인에 원문 저장하지 않는 설명, 조회용 메타데이터, DAO 목록 캐시, 화면 표시 보조 데이터 저장. MVP에서는 Cloudflare D1을 우선 검토하고, 요구사항상 필요하면 외부 Postgres와 Cloudflare Hyperdrive 또는 Prisma Accelerate/Data Proxy 사용을 허용 |
| Cloudflare R2 | 증빙 파일 원문 저장 |
| Workers Cron | Factory/DAO 이벤트 동기화 작업 주기 실행 |

## 3.2 전제 및 가정

- MVP는 Sepolia 테스트넷 ETH를 사용한다.
- 사용자는 MetaMask 등 EVM 호환 지갑을 보유한다.
- 조직 구성원은 지갑 주소로 식별한다.
- 각 DAO는 독립적인 금고 잔액과 제안 목록을 가진다.
- DAO 생성은 Factory 컨트랙트를 통해 수행되며, Factory는 조직별 DAO 금고 컨트랙트를 생성한다.
- 투표권은 조직 생성 시 등록된 구성원 주소에 부여된다.
- MVP에서는 DAO 생성 이후 구성원 추가/삭제 기능을 제공하지 않는다.
- 자금 집행 검증에 필요한 핵심 상태, 제안 상세 데이터 해시, 증빙 파일별 해시는 온체인에 저장하고, 그 외 상세 설명과 화면 표시 보조 데이터 원문은 서비스 DB에 저장한다.
- DAO 목록의 원본 데이터는 Factory 컨트랙트 이벤트와 온체인 상태를 기준으로 하며, 서비스 DB는 검색, 정렬, 빠른 조회를 위한 캐시로 사용한다.
- 온체인 데이터와 서비스 DB 캐시가 불일치하면 온체인 데이터를 기준으로 재동기화한다.
- MVP의 기본 배포 대상은 Cloudflare Pages와 Cloudflare Workers/Pages Functions이다.
- API 계층은 Cloudflare Workers 런타임에서 실행 가능한 HTTP handler 구조로 구현한다. Express 등 일반 Node.js 서버 프레임워크를 사용할 경우 Workers 런타임 호환성을 사전에 검증해야 한다.
- 서비스 DB는 Cloudflare D1을 우선 검토한다. Prisma 사용, 복잡한 쿼리, 마이그레이션, 트랜잭션 요구가 D1과 맞지 않을 경우 외부 Postgres와 Cloudflare Hyperdrive 또는 Prisma Accelerate/Data Proxy 사용을 허용한다.
- 증빙 파일 원문은 Cloudflare R2에 저장하고, 서비스 DB에는 object key, MIME type, 파일 크기, 파일 해시, 업로더 주소, 등록 시각만 저장한다.
- 온체인 이벤트 동기화는 Workers Cron Trigger로 주기 실행하며, `sync_state.last_synced_block` 이후 블록 범위를 조회해 DB 캐시에 반영한다.
- RPC URL, DB 접속 정보, R2 접근 정보 등 민감 설정은 Cloudflare Secrets로 관리한다.
- 컨트랙트 배포용 private key는 Cloudflare 런타임에 저장하지 않고, 로컬 또는 별도 CI/CD 배포 단계에서만 사용한다.

## 3.3 기능 요구사항

### FR-001 지갑 연결

- 사용자는 웹 애플리케이션에서 지갑 연결 버튼을 클릭할 수 있어야 한다.
- 시스템은 연결된 지갑 주소를 화면에 표시해야 한다.
- 시스템은 지원하지 않는 네트워크에 연결된 경우 네트워크 변경 안내를 표시해야 한다.

### FR-002 DAO 생성

- 사용자는 조직명을 입력할 수 있어야 한다.
- 사용자는 추가 구성원 지갑 주소를 등록할 수 있어야 한다.
- 시스템은 DAO 생성자의 지갑 주소를 구성원 목록에 자동 포함해야 한다.
- 시스템은 MVP에서 DAO 구성원 수를 최대 20명으로 제한해야 한다.
- 사용자는 기본 투표 기준으로 과반 찬성 또는 2/3 찬성을 선택할 수 있어야 한다.
- 시스템은 지갑 주소 형식이 유효하지 않으면 생성 요청을 막아야 한다.
- 시스템은 Factory 컨트랙트를 호출해 조직별 DAO 금고 컨트랙트를 생성해야 한다.
- 시스템은 Factory 컨트랙트가 반환하거나 이벤트로 기록한 DAO 금고 컨트랙트 주소를 저장하고 표시해야 한다.
- 시스템은 DAO 생성 트랜잭션이 완료되면 DAO 상세 화면으로 이동해야 한다.

### FR-003 DAO 목록 조회

- 지갑을 연결한 사용자는 본인이 구성원으로 등록된 DAO 목록만 조회할 수 있어야 한다.
- 시스템은 각 DAO의 조직명, 금고 잔액, 구성원 수, 진행 중 제안 수를 표시해야 한다.
- 시스템은 기본 목록에서 활성 DAO만 표시하고, 종료된 DAO는 종료 DAO 필터를 선택했을 때 표시해야 한다.
- 시스템은 DAO 목록의 원본 데이터를 Factory 컨트랙트 이벤트와 온체인 상태 기준으로 동기화해야 한다.
- 시스템은 서비스 DB를 DAO 목록 조회용 캐시로 사용할 수 있어야 한다.
- 시스템은 온체인 데이터와 서비스 DB 캐시가 불일치할 경우 온체인 데이터를 기준으로 재동기화해야 한다.
- 사용자는 DAO 항목을 선택해 상세 화면으로 이동할 수 있어야 한다.

### FR-004 DAO 상세 조회

- 사용자는 DAO의 기본 정보를 확인할 수 있어야 한다.
- 시스템은 조직명, DAO 금고 컨트랙트 주소, 구성원 수, 투표 기준, 현재 잔액을 표시해야 한다.
- 시스템은 최근 입금 내역, 최근 제안, 최근 집행 내역을 요약 표시해야 한다.

### FR-005 회비 입금

- DAO 구성원은 입금 금액을 입력할 수 있어야 한다.
- 시스템은 0 이하 금액 입력을 허용하지 않아야 한다.
- 사용자는 지갑에서 입금 트랜잭션을 승인할 수 있어야 한다.
- 시스템은 입금 완료 후 금고 잔액과 입금 내역을 갱신해야 한다.
- 시스템은 비구성원의 입금을 허용하지 않아야 한다.
- 시스템은 DAO가 종료 투표 중이거나 종료 상태인 경우 입금을 허용하지 않아야 한다.

### FR-006 입금 내역 조회

- DAO 구성원은 DAO별 입금 내역을 조회할 수 있어야 한다.
- 시스템은 입금자 주소, 입금 금액, 입금 시각, 트랜잭션 해시를 표시해야 한다.
- 시스템은 주소를 축약 표시하되 전체 주소 복사 기능을 제공해야 한다.

### FR-007 지출 제안 생성

- DAO 구성원은 지출 제안을 생성할 수 있어야 한다.
- 비구성원은 지출 제안 생성 화면에 접근하거나 제안 생성 트랜잭션을 실행할 수 없어야 한다.
- 제안에는 제목, 설명, 금액, 수신자 지갑 주소, 투표 마감일이 포함되어야 한다.
- 제안 생성자는 해당 제안에 한해 만장일치 조건을 선택할 수 있어야 한다.
- 시스템은 제안 제목, 설명, 금액, 수신자 주소 등 제안 상세 데이터를 canonical JSON으로 정규화하고 해시를 생성해 온체인 검증값으로 기록해야 한다.
- canonical JSON 해시 대상은 `schemaVersion`, `chainId`, `daoAddress`, `proposalType`, `proposer`, `title`, `description`, `amountWei`, `recipient`, `deadline`, `approvalType`을 포함해야 한다.
- 특정 제안 유형에서 사용하지 않는 필드는 canonical JSON에서 `null`로 정규화해야 한다. 예를 들어 DAO 종료 제안의 `amountWei`와 `recipient`는 `null`로 기록한다.
- canonical JSON은 UTF-8 기준, key 사전순 정렬, 불필요한 공백 제거, 금액은 wei 단위 문자열, 시간은 Unix timestamp, 지갑 주소는 소문자 주소 문자열로 정규화해야 한다.
- 제안 상세 데이터 해시는 `keccak256(UTF-8 canonical JSON bytes)` 방식으로 생성해야 한다.
- 시스템은 `proposalType`이 지출 제안인 경우 `amountWei`와 `recipient`를 필수로 검증해야 한다.
- 시스템은 금고 잔액보다 큰 금액도 제안 생성은 허용할 수 있으나, 집행 시 잔액 부족이면 실패 처리해야 한다.
- 시스템은 제목, 금액, 수신자 주소, 마감일이 없으면 등록을 막아야 한다.
- 시스템은 과거 시간을 투표 마감일로 설정하지 못하게 해야 한다.
- 시스템은 DAO가 종료 투표 중이거나 종료 상태인 경우 신규 지출 제안 생성을 허용하지 않아야 한다.

### FR-008 제안 목록 조회

- DAO 구성원은 DAO별 지출 제안 목록을 조회할 수 있어야 한다.
- 시스템은 제목, 금액, 수신자 주소, 상태, 찬성률, 마감일을 표시해야 한다.
- DAO 구성원은 상태별로 제안을 필터링할 수 있어야 한다.

### FR-009 제안 상세 조회

- DAO 구성원은 지출 제안의 상세 정보를 조회할 수 있어야 한다.
- 시스템은 제목, 설명, 금액, 수신자 주소, 생성자, 상태, 마감일을 표시해야 한다.
- 시스템은 찬성 수, 반대 수, 미투표 수, 승인 기준 충족 여부를 표시해야 한다.
- 시스템은 해당 제안의 투표 기록을 표시해야 한다.
- 취소된 제안인 경우 시스템은 취소자, 취소 시각, 취소 사유를 표시해야 한다.

### FR-010 투표

- DAO 구성원은 투표 중 상태의 제안에 찬성 또는 반대 투표를 할 수 있어야 한다.
- 시스템은 동일 주소가 동일 제안에 두 번 투표하지 못하게 해야 한다.
- 시스템은 비구성원의 투표를 허용하지 않아야 한다.
- 시스템은 `block.timestamp < deadline`인 경우에만 투표를 허용해야 한다.
- 시스템은 `block.timestamp >= deadline`인 경우 투표를 허용하지 않아야 한다.
- 시스템은 취소된 제안에 대한 투표를 허용하지 않아야 한다.
- 투표 완료 후 화면에 최신 투표 현황을 표시해야 한다.

### FR-011 집행 가능 여부 판단

- 시스템은 투표 마감 후 제안이 승인 조건을 만족하는지 판단할 수 있어야 한다.
- 승인 조건은 DAO 생성 시 설정한 기본 기준을 따르되, 제안 생성 시 만장일치가 선택된 경우 만장일치 기준을 우선 적용한다.
- 투표 마감 여부는 `block.timestamp >= deadline`으로 판단해야 한다.
- 과반 찬성은 `yesVotes * 2 > memberCount`로 판단해야 한다.
- 2/3 찬성은 `yesVotes * 3 >= memberCount * 2`로 판단해야 한다.
- 만장일치는 `yesVotes == memberCount`로 판단해야 한다.
- 투표 마감 후 제안이 승인 조건을 만족하면 집행 가능 상태로 표시해야 한다.
- 투표 마감 후 승인 조건을 만족하지 못하면 부결 상태로 표시해야 한다.
- 취소된 제안은 승인 조건 판단 대상에서 제외되어야 한다.

### FR-012 자동 집행

- 집행 가능 상태의 제안은 DAO 구성원이 실행 버튼을 눌러 집행할 수 있어야 한다.
- 비구성원은 집행 실행 트랜잭션을 실행할 수 없어야 한다.
- 스마트컨트랙트는 집행 시점에 승인 조건, 잔액, 중복 집행 여부를 검증해야 한다.
- 조건이 유효하면 금고에서 수신자 주소로 금액을 송금해야 한다.
- 집행 완료 후 제안 상태는 집행 완료로 변경되어야 한다.
- 권한 없음, 잘못된 제안 상태, 중복 집행, 취소된 제안, 종료 투표 중/종료 상태 등 사전 검증 실패는 트랜잭션을 revert해야 한다.
- 집행 가능 상태의 제안에서 컨트랙트 잔액 부족 또는 수신자 송금 실패가 발생하면 트랜잭션은 revert하지 않고 제안 상태를 집행 실패로 변경해야 한다.
- 집행 실패 상태로 변경된 제안은 다시 집행할 수 없어야 한다.
- 이미 집행된 제안은 다시 집행될 수 없어야 한다.
- 취소된 제안은 집행될 수 없어야 한다.
- DAO가 종료 투표 중이거나 종료 상태인 경우 일반 지출 제안은 집행될 수 없어야 한다.

### FR-013 예산 내역 조회

- DAO 구성원은 DAO별 예산 내역을 조회할 수 있어야 한다.
- 시스템은 입금 내역, 지출 제안 내역, 제안 취소 내역, 투표 기록, 집행 내역을 구분해 표시해야 한다.
- DAO 구성원은 각 내역의 트랜잭션 해시를 확인할 수 있어야 한다.
- 시스템은 현재 잔액을 항상 상단에 표시해야 한다.

### FR-014 공개 검증

- 사용자는 트랜잭션 해시 또는 DAO 금고 컨트랙트 주소를 통해 블록 익스플로러로 이동할 수 있어야 한다.
- 시스템은 온체인 기록과 화면 데이터가 불일치할 경우 새로고침 또는 재동기화 안내를 표시해야 한다.
- 비구성원에게는 웹 서비스 내 DAO 목록, 제안 상세, 증빙, 예산 상세를 제공하지 않아야 한다.
- 외부 검증은 컨트랙트 주소와 트랜잭션 링크를 통한 블록 익스플로러 확인으로 제한한다.

### FR-015 지출 증빙 등록 및 조회

- 제안자는 집행 완료된 본인 지출 제안에 대해 증빙 자료를 등록할 수 있어야 한다.
- DAO 종료 제안에는 증빙 자료를 등록할 수 없어야 한다.
- 제안자가 아닌 구성원은 증빙 자료를 등록할 수 없어야 한다.
- 증빙 자료에는 영수증, 구매 내역, 사진, 거래명세서, 설명 메모가 포함될 수 있다.
- 시스템은 증빙 자료를 온체인에 직접 저장하지 않고 Cloudflare R2에 저장해야 한다.
- 시스템은 증빙 자료 원문을 서비스 DB에 저장하지 않고, DB에는 R2 object key와 메타데이터만 저장해야 한다.
- 시스템은 증빙 자료의 파일별 `SHA-256` 해시를 생성해 온체인 검증값으로 기록해야 한다.
- 시스템은 DAO 금고 컨트랙트의 증빙 해시 등록 함수를 호출해 `proposalId`, 증빙 파일 해시, 등록자, 등록 시각을 기록해야 한다.
- 시스템은 등록된 증빙 자료의 삭제, 수정, 교체를 허용하지 않아야 한다.
- 시스템은 정정이나 보완이 필요한 경우 제안자가 추가 증빙을 새로 등록하는 방식만 허용해야 한다.
- 증빙 등록자는 온체인 해시 기록 트랜잭션의 가스비를 부담해야 한다.
- DAO 구성원은 제안 상세 화면에서 등록된 증빙 자료를 조회할 수 있어야 한다.
- 비구성원은 증빙 자료의 상세 내용을 조회할 수 없어야 한다.
- 시스템은 증빙 자료가 지출 목적을 자동 보증하는 것이 아니라 구성원이 사후 확인하기 위한 자료임을 표시해야 한다.

### FR-016 DAO 종료

- DAO 구성원은 DAO 종료 제안을 생성할 수 있어야 한다.
- 시스템은 진행 중 제안 또는 집행 가능 제안이 있는 경우 DAO 종료 제안 생성을 허용하지 않아야 한다.
- 종료 제안에는 남은 잔액을 전체 구성원에게 균등 분배한다는 내용이 포함되어야 한다.
- 종료 제안은 `proposalType`이 DAO 종료인 경우 `amountWei`와 `recipient`를 필수로 요구하지 않아야 한다.
- 종료 제안은 온체인 함수와 이벤트에서는 `amountWei = 0`, `recipient = address(0)`으로 기록하고, DB와 canonical JSON에서는 해당 필드를 `null`로 정규화해야 한다.
- 종료 제안은 일반 지출 제안과 동일하게 투표 마감 후 승인 조건을 만족해야 한다.
- 종료 투표 중에는 입금, 신규 지출 제안 생성, 일반 제안 집행을 차단하고 종료 제안 투표만 허용해야 한다.
- 종료 제안자는 투표 마감 전 종료 제안을 취소할 수 있어야 한다.
- 종료 제안이 취소되거나 부결되면 DAO 상태는 활성으로 복귀해야 한다.
- 종료 제안이 승인되면 DAO 구성원은 종료 실행 버튼을 누를 수 있어야 한다.
- 시스템은 종료 실행 시 DAO 금고의 남은 잔액을 DAO 구성원에게 균등 반환해야 한다.
- 시스템은 종료 실행 시 1인 반환액을 `DAO 잔액 / 구성원 수`로 계산해야 한다.
- 시스템은 균등 분배 후 남는 나머지 wei를 DAO 생성자에게 반환해야 한다.
- 시스템은 종료 실행 중 구성원 중 한 명에게라도 송금이 실패하면 종료 실행 전체를 실패 처리하고 DAO 상태를 유지해야 한다.
- 잔액 반환이 완료되면 DAO 상태는 종료로 변경되어야 한다.
- 종료된 DAO는 신규 입금, 지출 제안 생성, 투표, 집행을 허용하지 않아야 한다.

### FR-017 제안 취소

- 제안자는 본인이 생성한 제안을 투표 마감 전까지 취소할 수 있어야 한다.
- 제안 취소는 제안 상태가 투표 중인 경우에만 허용되어야 한다.
- 제안자는 제안 취소 시 취소 사유를 입력해야 한다.
- 시스템은 제안자가 아닌 사용자의 제안 취소를 허용하지 않아야 한다.
- 시스템은 투표 마감 후, 집행 가능, 부결, 집행 완료, 집행 실패, 취소됨 상태의 제안 취소를 허용하지 않아야 한다.
- 제안이 취소되면 상태는 취소됨으로 변경되어야 한다.
- 취소된 제안의 기존 투표 기록은 삭제하지 않고 조회 가능해야 한다.
- 취소된 제안은 추가 투표, 집행, 증빙 등록을 허용하지 않아야 한다.

## 3.4 비기능 요구사항

### NFR-001 보안

- 개인키는 웹 애플리케이션에서 저장하거나 입력받지 않는다.
- 모든 자금 이동은 지갑 서명과 스마트컨트랙트 검증을 통해서만 수행한다.
- 스마트컨트랙트는 중복 집행을 방지해야 한다.
- 스마트컨트랙트는 재진입 공격 가능성을 고려해 송금 로직을 설계해야 한다.
- 수신자 주소와 금액은 집행 직전에 다시 검증되어야 한다.

### NFR-002 투명성

- 입금, 제안, 투표, 집행 이벤트는 조회 가능해야 한다.
- 주요 트랜잭션은 블록 익스플로러 링크를 제공해야 한다.
- 사용자는 현재 잔액과 누적 입출금 흐름을 한 화면에서 이해할 수 있어야 한다.
- 지출 증빙 자료는 온체인 집행 기록과 함께 조회되어야 하며, 온체인 기록과 오프체인 증빙의 역할 차이가 명확히 표시되어야 한다.

### NFR-003 사용성

- 지갑 연결 전과 후의 가능한 행동을 명확히 구분해야 한다.
- 트랜잭션 대기, 성공, 실패 상태를 사용자에게 표시해야 한다.
- 긴 지갑 주소는 축약 표시하되 복사할 수 있어야 한다.
- 금액 단위는 일관되게 표시해야 한다.

### NFR-004 성능

- DAO 상세 화면의 주요 정보는 3초 이내에 표시되는 것을 목표로 한다.
- 트랜잭션 완료 시간은 블록체인 네트워크 상태에 의존하므로 대기 상태를 명확히 표시한다.
- 목록 화면은 페이지네이션 또는 필터링을 고려한다.

### NFR-005 호환성

- MVP는 EVM 호환 테스트넷을 기준으로 한다.
- 데스크톱 브라우저 환경을 우선 지원한다.
- MetaMask 또는 호환 지갑 연결을 전제로 한다.
- MVP 운영 배포는 Cloudflare Pages와 Cloudflare Workers/Pages Functions를 기준으로 한다.
- Workers 런타임에서 지원하지 않는 Node.js API, 장기 실행 서버, 로컬 파일 시스템 의존 기능은 사용하지 않는다.
- API 구현에 Node.js 패키지를 사용할 경우 Workers 런타임 호환성을 확인한 뒤 도입한다.
- 온체인 이벤트 동기화는 Workers Cron 실행 시간과 RPC provider rate limit을 고려해 블록 범위를 나누어 처리해야 한다.

### NFR-006 감사 가능성

- 모든 주요 상태 변경은 이벤트 로그로 남겨야 한다.
- 제안 생성, 제안 취소, 투표, 집행 이벤트는 추후 화면에서 재구성 가능해야 한다.
- 관리자가 임의로 내역을 수정할 수 없어야 한다.
- 등록된 증빙 자료는 삭제, 수정, 교체할 수 없어야 하며, 정정이나 보완은 추가 증빙 등록으로만 처리해야 한다.

## 3.6 스마트컨트랙트 요구사항

### SC-001 Factory 기반 DAO 생성

- Factory 컨트랙트는 DAO 생성 요청을 받아 조직별 DAO 금고 컨트랙트를 배포해야 한다.
- DAO 생성 시 조직명, 구성원 주소 목록, 승인 기준, 생성자 주소를 DAO 금고 컨트랙트에 저장해야 한다.
- DAO 생성자의 지갑 주소는 구성원 목록에 자동 포함되어야 한다.
- DAO 생성자는 영구 운영자로 기록되어야 한다.
- Factory 컨트랙트는 생성된 DAO 금고 컨트랙트 주소를 기록해야 한다.
- Factory 컨트랙트는 DAO 생성 이벤트를 발생시켜야 한다.
- 중복 구성원 주소는 허용하지 않아야 한다.
- 구성원 수가 0명인 DAO는 생성할 수 없어야 한다.
- MVP에서는 DAO 구성원 수가 20명을 초과할 수 없어야 한다.

### SC-002 금고 입금

- DAO 금고 컨트랙트는 회비 입금을 받을 수 있어야 한다.
- 입금 발생 시 입금자와 금액 이벤트를 발생시켜야 한다.
- 구성원 주소만 입금할 수 있어야 한다.
- DAO가 종료 투표 중이거나 종료 상태이면 입금을 차단해야 한다.

### SC-003 제안 생성

- 구성원만 제안을 생성할 수 있어야 한다.
- 제안 생성 시 제안 유형, 금액, 수신자, 마감일, 승인 기준 유형, canonical JSON 기반 제안 상세 데이터 해시를 저장해야 한다.
- 지출 제안은 금액과 수신자를 필수로 검증해야 한다.
- DAO 종료 제안은 금액과 수신자를 필수로 요구하지 않고, 종료 정책에 따라 남은 잔액을 구성원에게 균등 분배해야 한다.
- DAO 종료 제안은 온체인 저장값과 이벤트에서 `amountWei = 0`, `recipient = address(0)`으로 기록해야 한다.
- DAO가 종료 투표 중이거나 종료 상태이면 신규 지출 제안 생성을 차단해야 한다.
- canonical JSON은 key 사전순 정렬, wei 단위 금액 문자열, Unix timestamp, 소문자 주소 문자열 규칙을 따라야 한다.
- 제안 상세 데이터 해시는 `keccak256(UTF-8 canonical JSON bytes)` 방식으로 생성해야 한다.
- 제안 생성 이벤트를 발생시켜야 한다.

### SC-004 투표

- 구성원만 투표할 수 있어야 한다.
- 제안별 주소당 1회만 투표할 수 있어야 한다.
- `block.timestamp < deadline`인 경우에만 투표할 수 있어야 한다.
- `block.timestamp >= deadline`이면 투표할 수 없어야 한다.
- 취소된 제안에는 투표할 수 없어야 한다.
- 투표 이벤트를 발생시켜야 한다.

### SC-005 집행

- 승인 조건을 만족한 제안만 집행할 수 있어야 한다.
- 승인 조건 계산은 과반 찬성 `yesVotes * 2 > memberCount`, 2/3 찬성 `yesVotes * 3 >= memberCount * 2`, 만장일치 `yesVotes == memberCount` 기준을 사용해야 한다.
- 제안 결과 확정과 집행 가능 여부 판단은 `block.timestamp >= deadline`인 경우에만 가능해야 한다.
- 구성원만 집행을 실행할 수 있어야 한다.
- 모든 지출은 사전에 생성된 제안과 투표 승인 결과에 연결되어야 한다.
- 이미 집행된 제안은 집행할 수 없어야 한다.
- 취소된 제안은 집행할 수 없어야 한다.
- DAO가 종료 투표 중이거나 종료 상태이면 일반 지출 제안은 집행할 수 없어야 한다.
- 권한 없음, 잘못된 제안 상태, 중복 집행, 취소된 제안, 종료 투표 중/종료 상태 등 사전 검증 실패는 revert해야 한다.
- 집행 가능 상태에서 컨트랙트 잔액이 부족하거나 수신자 송금이 실패하면 revert하지 않고 제안 상태를 집행 실패로 변경해야 한다.
- 집행 실패 상태의 제안은 다시 집행할 수 없어야 한다.
- 집행 완료 시 수신자에게 금액을 송금하고 집행 이벤트를 발생시켜야 한다.
- 집행 실패 시 집행 실패 이벤트를 발생시켜야 한다.

### SC-006 DAO 종료

- 구성원만 DAO 종료 제안을 생성할 수 있어야 한다.
- 진행 중 제안 또는 집행 가능 제안이 있으면 DAO 종료 제안을 생성할 수 없어야 한다.
- 종료 제안은 남은 잔액을 전체 구성원에게 균등 분배하는 방식이어야 한다.
- 일반 지출 제안과 동일한 승인 조건을 만족한 종료 제안만 실행할 수 있어야 한다.
- 종료 투표 중에는 입금, 신규 지출 제안 생성, 일반 제안 집행을 차단하고 종료 제안 투표만 허용해야 한다.
- 종료 제안자는 투표 마감 전 종료 제안을 취소할 수 있어야 한다.
- 종료 제안이 취소되거나 부결되면 DAO 상태를 활성으로 변경해야 한다.
- 종료 실행 시 DAO 금고 잔액을 DAO 구성원에게 균등 반환해야 한다.
- 종료 실행 시 1인 반환액은 `DAO 잔액 / 구성원 수`로 계산해야 한다.
- 균등 분배 후 남는 나머지 wei는 DAO 생성자에게 반환해야 한다.
- 구성원 중 한 명에게라도 송금이 실패하면 종료 실행 전체를 실패 처리하고 DAO 상태를 유지해야 한다.
- 잔액 반환 완료 후 DAO 상태를 종료로 변경해야 한다.
- 종료 상태에서는 입금, 제안 생성, 투표, 집행을 차단해야 한다.
- 종료 이벤트를 발생시켜야 한다.

### SC-007 제안 취소

- 제안자만 본인이 생성한 제안을 취소할 수 있어야 한다.
- 투표 마감 전이고 투표 중 상태인 제안만 취소할 수 있어야 한다.
- 제안 취소 사유 원문은 서비스 DB에 저장하고, 취소 사유 해시는 `keccak256(UTF-8 cancel reason bytes)` 방식으로 온체인에 기록해야 한다.
- 제안 취소 후 제안 상태를 취소됨으로 변경해야 한다.
- 취소된 제안은 투표, 집행, 증빙 등록을 차단해야 한다.
- 제안 취소 이벤트를 발생시켜야 한다.

### SC-008 증빙 해시 등록

- DAO 금고 컨트랙트는 집행 완료된 지출 제안에 대해 증빙 해시를 등록할 수 있는 함수를 제공해야 한다.
- DAO 종료 제안에는 증빙 해시를 등록할 수 없어야 한다.
- 제안자만 본인 지출 제안의 증빙 해시를 등록할 수 있어야 한다.
- 증빙 파일 원문은 온체인에 저장하지 않아야 한다.
- 증빙 파일 해시는 `SHA-256` 값을 사용해야 한다.
- 컨트랙트는 `proposalId`, 증빙 파일 해시, 등록자 주소, 등록 시각을 기록해야 한다.
- 증빙 해시 등록 이벤트를 발생시켜야 한다.

## 4.2 핵심 메시지

- 이 서비스는 단순 투표가 아니라 돈이 걸린 공동 금고이다.
- 운영자가 혼자 돈을 쓸 수 없다는 점이 핵심이다.
- 회비 입금부터 지출 집행까지 모든 흐름이 검증 가능하다.
- 승인 조건을 코드로 고정하고 스마트컨트랙트가 자동 집행한다.
- MVP에서는 테스트넷 자산으로 구현해 법적, 결제 복잡도를 줄인다.

## 6. 확정된 정책 요약

아래 항목은 기존 검토사항에 대해 확정된 결정이다.

| 항목 | 확정 내용 |
|---|---|
| 종료 제안 데이터 구조 | `proposalType`별로 필수 필드를 다르게 검증한다. 지출 제안은 `amountWei`와 `recipient`가 필수이고, DAO 종료 제안은 해당 필드를 필수로 요구하지 않는다. |
| DAO 종료 시 잔액 반환 방식 | MVP에서는 구성원 수를 최대 20명으로 제한하고, 종료 실행 시 컨트랙트가 모든 구성원에게 즉시 균등 송금한다. 1인 반환액은 `DAO 잔액 / 구성원 수`로 계산하고, 나머지 wei는 DAO 생성자에게 반환한다. |
| 종료 투표 생성 조건 | 진행 중 제안과 집행 가능 제안이 없을 때만 DAO 종료 제안을 생성할 수 있다. |
| 종료 투표 중 동작 정책 | 종료 투표 중에는 입금, 신규 지출 제안 생성, 일반 제안 집행을 차단하고 종료 제안 투표만 허용한다. 종료 제안자는 투표 마감 전 종료 제안을 취소할 수 있다. |
| 종료 제안 취소/부결 시 DAO 상태 | 종료 제안이 취소되거나 부결되면 DAO 상태를 활성으로 복귀시킨다. |
| 종료 즉시 송금 실패 처리 | 종료 실행 중 구성원 중 한 명에게라도 송금이 실패하면 종료 실행 전체를 실패 처리하고 DAO 상태를 유지한다. |
| 증빙 해시 등록 방식 | DAO 금고 컨트랙트에 증빙 해시 등록 함수를 추가한다. 증빙 파일 원문은 온체인에 저장하지 않고, 증빙 파일 해시와 관련 메타데이터만 기록한다. |
| 증빙 등록 대상 제안 범위 | 증빙 등록은 집행 완료된 지출 제안에만 허용한다. DAO 종료 제안에는 증빙 자료나 증빙 해시를 등록할 수 없다. |
| canonical JSON 해시 표준 | 제안 상세 데이터는 `schemaVersion`, `chainId`, `daoAddress`, `proposalType`, `proposer`, `title`, `description`, `amountWei`, `recipient`, `deadline`, `approvalType`을 포함한 canonical JSON으로 정규화한 뒤 해시를 생성한다. key는 사전순으로 정렬하고, 금액은 wei 단위 문자열, 시간은 Unix timestamp, 지갑 주소는 소문자 주소 문자열로 통일한다. 특정 제안 유형에서 사용하지 않는 필드는 `null`로 정규화한다. |
| 해시 알고리즘 | 제안 상세 데이터는 `keccak256(UTF-8 canonical JSON bytes)`를 사용하고, 증빙 파일은 `SHA-256`을 사용한다. |
| 취소 사유 해시 알고리즘 | 취소 사유 해시는 `keccak256(UTF-8 cancel reason bytes)`를 사용한다. |
| 증빙 자료 수정 및 교체 정책 | 등록된 증빙 자료는 삭제, 수정, 교체할 수 없고, 정정이나 보완이 필요한 경우 추가 증빙만 등록한다. |
| 외부 검증 링크 및 조회 범위 | 외부 검증은 컨트랙트 주소와 트랜잭션 해시 기반 블록 익스플로러 링크만 제공한다. 별도 외부 검증 화면은 MVP 범위에서 제외한다. |
| DAO 종료 제안 화면 | 일반 지출 제안 화면을 재사용하지 않고, DAO 종료 제안 생성 화면을 별도로 제공한다. |
| SC-003 제안 생성 범위 | SC-003은 공통 제안 생성 요구사항으로 두고, `proposalType`에 따라 지출 제안과 DAO 종료 제안의 필수 필드와 검증 조건을 다르게 명시한다. |
| enum 숫자값 기준 | 스마트컨트랙트, 이벤트, DB, 프론트엔드가 동일한 숫자값 매핑을 사용한다. 코드에서는 숫자를 직접 사용하지 않고 상수명 또는 enum 이름을 함께 사용한다. |
| DAO 종료 제안 온체인 표현 | DAO 종료 제안은 온체인 함수와 이벤트에서 `amountWei = 0`, `recipient = address(0)`으로 기록한다. DB와 canonical JSON에서는 사용하지 않는 필드를 `null`로 정규화한다. |
| DAO 생성 구성원 파라미터 | `createDAO`의 구성원 배열은 생성자를 제외한 추가 구성원 주소 목록으로 정의한다. 생성자는 컨트랙트에서 자동으로 구성원에 포함한다. |
| 제안 결과 확정 권한 | `finalizeProposal`은 누구나 실행 가능한 상태 변경 함수로 정의한다. 조회는 `getProposal`이 담당한다. |
| 지출 집행 실패 처리 | 권한, 상태, 중복 집행 등 사전 검증 실패는 revert한다. 집행 가능 상태에서 잔액 부족 또는 수신자 송금 실패가 발생하면 revert하지 않고 제안 상태를 `집행 실패`로 변경하고 실패 이벤트를 기록한다. |
| 투표 승인 계산식 | 과반 찬성은 `yesVotes * 2 > memberCount`, 2/3 찬성은 `yesVotes * 3 >= memberCount * 2`, 만장일치는 `yesVotes == memberCount`로 계산한다. |
| 투표 마감 경계 조건 | 투표는 `block.timestamp < deadline`일 때만 허용하고, `block.timestamp >= deadline`이면 투표 마감으로 판단해 결과 확정과 집행 가능 여부 판단을 허용한다. |
| 집행 실패 사유 코드 | `ProposalExecutionFailed.reasonCode`는 `1 = INSUFFICIENT_BALANCE`, `2 = TRANSFER_FAILED`로 정의한다. |

## 7.5 온체인/DB 분리 기준

| 데이터 유형 | 저장 위치 | 이유 |
|---|---|---|

| 제안 제목/설명 원문 | DB | 원문은 길고 비용이 크므로 해시만 온체인 저장 |
| 제안 상세 데이터 해시 | 온체인 | DB 원문 위변조 검증 |
| 취소 사유 원문 | DB | 원문 저장 비용과 개인정보/민감 정보 가능성 고려 |
| 취소 사유 해시 | 온체인 | 취소 사유 원문 위변조 검증 |
| 증빙 파일 원문 | Cloudflare R2 | 파일 원문은 온체인 저장 불가이며 DB에는 저장하지 않음 |
| 증빙 파일 SHA-256 해시 | 온체인 | 파일 위변조 검증 |
| DAO 목록 캐시 | DB | 빠른 검색, 정렬, 화면 표시 |
| Factory/DAO 이벤트 | 온체인 원본, DB 캐시 | 온체인을 원본으로 보고 DB는 조회 최적화 |

## 7.6 Cloudflare 배포 및 운영 기준

| 영역 | 기준 |
|---|---|
| 프론트엔드 | Cloudflare Pages에 Vite/React 정적 빌드 결과를 배포 |
| API | Cloudflare Workers 또는 Pages Functions에서 HTTP handler로 구현 |
| DB | Cloudflare D1 우선. 필요 시 외부 Postgres와 Hyperdrive 또는 Prisma Accelerate/Data Proxy 허용 |
| 파일 저장 | Cloudflare R2에 증빙 파일 원문 저장 |
| 이벤트 동기화 | Workers Cron으로 Factory/DAO 이벤트를 주기 동기화 |
| 비밀값 | Cloudflare Secrets로 RPC URL, DB URL, R2 접근 정보 관리 |
| 컨트랙트 배포 키 | Cloudflare 런타임에 저장하지 않고 로컬 또는 CI/CD 배포 단계에서만 사용 |
| 런타임 제약 | Workers 런타임에서 지원하지 않는 Node.js API와 장기 실행 Express 서버 의존을 금지 |

이벤트 동기화 작업은 마지막 성공 블록을 `sync_state`에 저장하고, 실패 시 다음 Cron 실행에서 마지막 성공 블록 이후부터 재시도한다. RPC provider rate limit을 고려해 한 번에 조회하는 블록 범위와 DAO 컨트랙트 수를 제한한다.

## 8. 용어 정의

| 용어 | 정의 |
|---|---|
| DAO | 구성원이 공동 의사결정으로 운영하는 탈중앙 조직 |
| Factory 컨트랙트 | DAO 생성 요청을 받아 조직별 DAO 금고 컨트랙트를 배포하고 생성된 DAO 주소를 기록하는 컨트랙트 |
| 스마트컨트랙트 금고 | 회비를 보관하고 조건 충족 시 송금하는 온체인 계약 |
| 제안 | 특정 수신자에게 특정 금액을 지출하자는 안건 |
| 투표 기준 | 제안 승인 여부를 판단하는 찬성 조건 |
| 집행 | 승인된 제안에 따라 금고에서 수신자에게 금액을 송금하는 행위 |
| 테스트넷 | 실제 가치가 없는 테스트용 블록체인 네트워크 |
