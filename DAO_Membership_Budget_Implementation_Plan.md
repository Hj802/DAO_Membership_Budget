# DAO 회비/예산 집행 구현 계획서

## 1. 문서 목적

본 문서는 `DAO_Membership_Budget_PRD_SRS.md`를 기준으로 1인 개발자가 MVP를 구현하기 위한 단계별 실행 계획이다.

각 Phase는 1 Man Day 기준으로 작성하며, 모든 Phase는 다음 활동을 포함한다.

- 구현
- 코드 리뷰
- 검증
- 테스트
- 사이드 이펙트 검증
- 지금까지 구현된 모든 Phase 누적 검증

## 2. 구현 원칙

- 스마트컨트랙트가 자금 이동과 핵심 상태의 원본이다.
- 서비스 DB는 온체인 이벤트 기반 조회 캐시와 오프체인 상세 데이터 저장소로 사용한다.
- 프론트엔드는 컨트랙트 상태와 DB 데이터를 함께 보여주되, 충돌 시 온체인을 기준으로 한다.
- MVP는 Sepolia 테스트넷 ETH 기준으로 구현한다.
- MVP 배포 대상은 Cloudflare Pages와 Cloudflare Workers/Pages Functions를 기준으로 한다.
- 증빙 파일 원문은 Cloudflare R2에 저장하고, DB에는 R2 object key와 메타데이터만 저장한다.
- 온체인 이벤트 동기화는 Workers Cron으로 주기 실행할 수 있는 구조로 구현한다.
- Cloudflare Workers 런타임과 충돌하는 장기 실행 Express 서버, 로컬 파일 시스템 의존, 미검증 Node.js API 사용은 피한다.
- 1인 개발 흐름을 고려해, 뒤 Phase가 앞 Phase 산출물에 의존하도록 순서를 고정한다.
- 각 Phase 종료 시점에는 최소한 해당 Phase 범위의 기능이 깨지지 않는 수준의 테스트와 회귀 검증을 수행한다.

## 3. Phase 계획

## Phase 1. 프로젝트 골격 및 개발 환경 구성

### 목표

개발을 시작할 수 있는 기본 구조, 실행 명령, 환경 변수, 네트워크 설정을 준비한다.

### 구현

- 프론트엔드, 스마트컨트랙트, Cloudflare Workers/Pages Functions API, DB, R2 관련 디렉터리 구조 정의
- 패키지 매니저와 실행 스크립트 구성
- Sepolia 네트워크 설정 구조 준비
- Cloudflare 배포 환경을 고려한 환경 변수 예시 파일 정의
- RPC URL, DB URL, R2 설정, 컨트랙트 주소, 배포용 private key의 책임 범위 분리
- Workers 런타임 호환성 확인을 위한 API 패키지 구조 준비
- lint, format, test 명령 구성
- 기본 README 실행 절차 초안 작성

### 코드 리뷰

- 디렉터리 구조가 역할별로 분리되어 있는지 확인
- 환경 변수에 개인키, API 키 같은 민감 정보가 커밋되지 않도록 확인
- 컨트랙트 배포용 private key가 Cloudflare 런타임 변수로 들어가지 않도록 확인
- API 골격이 장기 실행 Express 서버에 강하게 묶이지 않았는지 확인
- 실행 명령이 1인 개발자가 반복 사용하기 쉬운지 확인

### 검증

- 로컬에서 전체 프로젝트 설치 가능 여부 확인
- 기본 빌드 명령 실행 확인
- 기본 테스트 명령 실행 확인
- Cloudflare Pages/Workers로 옮길 때 필요한 환경 변수 목록이 빠지지 않았는지 확인

### 테스트

- 빈 테스트 또는 샘플 테스트가 정상 실행되는지 확인
- lint/format 명령이 실패 없이 동작하는지 확인
- API 샘플 handler가 Workers 런타임 제약과 충돌하지 않는지 확인

### 사이드 이펙트 검증

- 생성 파일이 PRD/SRS 문서와 충돌하지 않는지 확인
- 민감 정보가 파일에 포함되지 않았는지 확인
- 로컬 개발 편의 설정이 Cloudflare 배포 설정과 충돌하지 않는지 확인

### 누적 검증

- Phase 1 산출물만 기준으로 전체 설치, 빌드, 테스트 명령을 다시 실행한다.

### 완료 기준

- 개발자가 한 명령 흐름으로 설치, 빌드, 테스트를 실행할 수 있다.

## Phase 2. 스마트컨트랙트 기본 구조 구현

### 목표

Factory 컨트랙트와 DAO 금고 컨트랙트의 기본 생성 흐름을 구현한다.

### 구현

- Factory 컨트랙트 구현
- DAO 금고 컨트랙트 기본 구조 구현
- `createDAO(name, additionalMembers, approvalRule)` 구현
- 생성자 자동 구성원 포함 처리
- 구성원 중복 검증
- 최대 구성원 수 20명 제한
- DAO 상태 enum, 승인 기준 enum, 제안 상태 enum 정의
- `DAOCreated` 이벤트 구현
- `getDAOsByMember`, `getAllDAOs`, `getMembers` 조회 함수 구현

### 코드 리뷰

- `additionalMembers`가 생성자를 제외한 목록으로 처리되는지 확인
- 생성자 자동 포함과 중복 검증 순서 확인
- enum 숫자값이 PRD/SRS의 확정값과 일치하는지 확인
- Factory가 생성 DAO 주소를 안정적으로 기록하는지 확인

### 검증

- DAO 생성 시 생성자가 구성원에 포함되는지 확인
- 구성원 수 20명 초과 시 실패하는지 확인
- 중복 주소 입력 시 실패하는지 확인
- Factory 이벤트와 조회 함수 결과가 일치하는지 확인

### 테스트

- DAO 생성 성공 테스트
- 생성자 자동 포함 테스트
- 중복 구성원 실패 테스트
- 최대 구성원 수 초과 실패 테스트
- `getDAOsByMember` 조회 테스트

### 사이드 이펙트 검증

- DAO 여러 개 생성 시 Factory 상태가 꼬이지 않는지 확인
- 한 주소가 여러 DAO에 속할 수 있는지 확인
- 비구성원이 DAO를 생성할 수 있는 정책과 충돌하지 않는지 확인

### 누적 검증

- Phase 1의 설치, 빌드, 테스트 명령을 다시 실행한다.
- Phase 2 컨트랙트 테스트 전체를 실행한다.

### 완료 기준

- Factory 기반 DAO 생성과 구성원 조회가 테스트로 검증된다.

## Phase 3. 입금 기능 및 금고 잔액 이벤트 구현

### 목표

DAO 구성원만 회비를 입금할 수 있고, 입금 내역을 이벤트로 추적할 수 있게 한다.

### 구현

- DAO 금고 `deposit()` payable 함수 구현
- 구성원만 입금 가능하도록 검증
- DAO 상태가 활성일 때만 입금 가능하도록 검증
- `DepositReceived` 이벤트 구현
- 현재 잔액 조회 방식 정리

### 코드 리뷰

- 비구성원 입금 차단 여부 확인
- 종료 투표 중/종료 상태에서 입금 차단 여부 확인
- 입금 이벤트 필드가 PRD/SRS와 일치하는지 확인

### 검증

- 구성원 입금 성공 확인
- 비구성원 입금 실패 확인
- 0 ETH 입금 처리 정책 확인
- 이벤트의 `balanceAfter`가 실제 컨트랙트 잔액과 일치하는지 확인

### 테스트

- 구성원 입금 성공 테스트
- 비구성원 입금 실패 테스트
- 종료 상태 입금 실패 테스트
- 이벤트 필드 검증 테스트

### 사이드 이펙트 검증

- 입금 후 Factory 조회 기능이 영향받지 않는지 확인
- 여러 구성원이 입금해도 잔액이 누적되는지 확인

### 누적 검증

- Phase 1~2 테스트 전체 실행
- Phase 3 입금 테스트 실행
- DAO 생성 후 입금까지 연결 시나리오 실행

### 완료 기준

- DAO 생성 후 구성원 입금과 이벤트 기록이 테스트로 검증된다.

## Phase 4. 제안 생성 및 제안 취소 구현

### 목표

지출 제안과 DAO 종료 제안을 공통 제안 구조로 생성하고, 제안자가 투표 마감 전 취소할 수 있게 한다.

### 구현

- `createProposal(proposalType, amountWei, recipient, deadline, approvalType, contentHash)` 구현
- 지출 제안 필수값 검증
- DAO 종료 제안의 `amountWei = 0`, `recipient = address(0)` 검증
- 종료 투표 생성 조건 검증
- 종료 투표 중 DAO 상태 전환 구현
- `ProposalCreated` 이벤트 구현
- `cancelProposal(proposalId, cancelReasonHash)` 구현
- 취소 권한, 상태, 마감 시간 검증
- `ProposalCanceled` 이벤트 구현

### 코드 리뷰

- 지출 제안과 종료 제안의 필수 필드 분기가 명확한지 확인
- 종료 제안 생성 시 진행 중 제안/집행 가능 제안 검사가 맞는지 확인
- 취소 사유 해시만 온체인에 저장되는지 확인
- 취소된 제안이 투표/집행/증빙 대상에서 제외되는지 확인

### 검증

- 지출 제안 생성 성공 확인
- 종료 제안 생성 성공 확인
- 종료 제안 생성 시 DAO 상태가 종료 투표 중으로 바뀌는지 확인
- 제안자만 취소 가능한지 확인
- 투표 마감 후 취소가 불가능한지 확인

### 테스트

- 지출 제안 생성 테스트
- 종료 제안 생성 테스트
- 종료 제안의 `0/address(0)` 검증 테스트
- 제안 취소 성공 테스트
- 비제안자 취소 실패 테스트
- 마감 후 취소 실패 테스트

### 사이드 이펙트 검증

- 종료 투표 중 입금, 신규 지출 제안 생성, 일반 집행이 차단되는지 확인
- 취소된 종료 제안이 DAO 상태를 활성으로 되돌리는지 확인
- 제안 취소가 기존 투표 기록 삭제를 유발하지 않는지 확인

### 누적 검증

- Phase 1~3 테스트 전체 실행
- DAO 생성, 입금, 제안 생성, 제안 취소 연결 시나리오 실행

### 완료 기준

- 지출/종료 제안 생성과 취소 정책이 컨트랙트 테스트로 검증된다.

## Phase 5. 투표 및 결과 확정 구현

### 목표

DAO 구성원의 1인 1표 투표와 승인 조건 계산을 구현한다.

### 구현

- `vote(proposalId, support)` 구현
- 구성원만 투표 가능하도록 검증
- 주소당 1회 투표 제한 구현
- `block.timestamp < deadline` 투표 가능 조건 구현
- 취소된 제안 투표 차단
- `VoteCast` 이벤트 구현
- `finalizeProposal(proposalId)` 구현
- `block.timestamp >= deadline` 결과 확정 조건 구현
- 과반, 2/3, 만장일치 수식 구현
- 부결/집행 가능 상태 전환 구현
- `ProposalFinalized` 이벤트 구현

### 코드 리뷰

- 승인 수식이 문서와 정확히 일치하는지 확인
- 전체 구성원 기준으로 계산하는지 확인
- 투표 참여자 기준으로 계산하지 않는지 확인
- 마감 경계 조건이 `< deadline`, `>= deadline`으로 정확히 나뉘는지 확인

### 검증

- 과반 승인/부결 케이스 확인
- 2/3 승인/부결 케이스 확인
- 만장일치 승인/부결 케이스 확인
- 투표 마감 전 결과 확정 실패 확인
- 투표 마감 후 투표 실패 확인

### 테스트

- 1인 1표 테스트
- 중복 투표 실패 테스트
- 비구성원 투표 실패 테스트
- 마감 전 finalize 실패 테스트
- 과반/2/3/만장일치 수식 테스트
- 취소된 제안 finalize 제외 테스트

### 사이드 이펙트 검증

- 투표가 입금 잔액에 영향을 주지 않는지 확인
- finalize가 송금을 수행하지 않는지 확인
- 누구나 finalize 가능하되 자금 집행 권한과 분리되어 있는지 확인

### 누적 검증

- Phase 1~4 테스트 전체 실행
- DAO 생성, 입금, 제안 생성, 투표, 결과 확정 연결 시나리오 실행

### 완료 기준

- 모든 승인 기준과 마감 조건이 테스트로 검증된다.

## Phase 6. 지출 집행 및 집행 실패 처리 구현

### 목표

승인된 지출 제안을 자동 송금하고, 실패 상태를 온체인에 남긴다.

### 구현

- `executeProposal(proposalId)` 구현
- DAO 구성원만 집행 가능하도록 검증
- 집행 가능 상태만 집행 허용
- 중복 집행 방지
- 종료 투표 중/종료 상태의 일반 지출 집행 차단
- 정상 송금 시 `ProposalExecuted` 이벤트 발생
- 잔액 부족 시 revert하지 않고 집행 실패 상태 전환
- 수신자 송금 실패 시 revert하지 않고 집행 실패 상태 전환
- `ProposalExecutionFailed` 이벤트 구현
- `reasonCode` 1, 2 상수 적용

### 코드 리뷰

- 사전 검증 실패와 실행 실패의 처리 방식이 분리되어 있는지 확인
- 잔액 부족/송금 실패만 집행 실패 상태로 남기는지 확인
- 재진입 공격 방지 패턴이 적용되었는지 확인
- 집행 완료 후 재집행이 불가능한지 확인

### 검증

- 정상 집행 후 수신자 잔액 증가 확인
- 정상 집행 후 제안 상태가 집행 완료인지 확인
- 잔액 부족 집행 시 제안 상태가 집행 실패인지 확인
- 송금 실패 수신자 케이스에서 실패 이벤트가 남는지 확인

### 테스트

- 정상 집행 테스트
- 비구성원 집행 실패 테스트
- 중복 집행 실패 테스트
- 잔액 부족 집행 실패 상태 기록 테스트

## Phase 8. Cloudflare DB/R2 스키마 및 이벤트 동기화 기반 구현

### 목표

Cloudflare 배포를 전제로 서비스 DB, R2 증빙 저장 메타데이터, 온체인 이벤트 동기화 기반을 구현한다.

### 구현

- Cloudflare D1 우선 구조로 DB 스키마 구현
- D1 사용이 어렵거나 Prisma 요구사항과 충돌할 경우 외부 Postgres + Hyperdrive 또는 Prisma Accelerate/Data Proxy 대안 경로 문서화
- `dao_cache` 테이블 구현
- `dao_members_cache` 테이블 구현
- `proposal_details` 테이블 구현
- `proposal_cancel_details` 테이블 구현
- `evidence_files` 테이블 구현
- `transaction_logs` 테이블 구현
- `sync_state` 테이블 구현
- `evidence_files`에는 파일 원문이 아니라 `r2_object_key`, `mime_type`, `file_size`, `content_hash` 저장
- R2 bucket binding 이름과 object key 규칙 정의
- Factory 이벤트 파싱 및 DB 반영 로직 구현
- DAO 금고 이벤트 파싱 및 DB 반영 로직 구현
- Workers Cron에서 호출 가능한 이벤트 동기화 entrypoint 구현
- `sync_state.last_synced_block` 기반 블록 범위 동기화 구현
- 온체인 우선 재동기화 정책 구현
- RPC provider rate limit을 고려한 블록 범위 제한 구현

### 코드 리뷰

- DB 컬럼이 PRD/SRS의 DB 저장 데이터와 일치하는지 확인
- 종료 제안의 `amount_wei`, `recipient` NULL 허용 여부 확인
- 증빙 파일 원문이 DB에 저장되지 않는지 확인
- R2 object key가 DAO/proposal/evidence 단위로 충돌 없이 생성되는지 확인
- 이벤트 중복 반영 방지 로직 확인
- 마지막 동기화 블록 관리 방식 확인
- Cron entrypoint가 한 번 실행되고 종료되는 작업 구조인지 확인
- Workers 런타임에서 지원하지 않는 Node.js API를 사용하지 않는지 확인

### 검증

- DAO 생성 이벤트가 DB에 반영되는지 확인
- 입금/제안/투표/집행/취소/증빙/종료 이벤트가 `transaction_logs`에 반영되는지 확인
- 온체인과 DB 불일치 시 재동기화가 가능한지 확인
- `sync_state`의 마지막 성공 블록 이후부터 재시도되는지 확인
- R2 object key와 DB 메타데이터가 연결 가능한지 확인

### 테스트

- 마이그레이션 테스트
- 이벤트 파싱 단위 테스트
- 중복 이벤트 처리 테스트
- 재동기화 테스트
- 종료 제안 NULL 필드 저장 테스트
- R2 object key 생성 테스트
- Cron 동기화 entrypoint 테스트
- RPC 조회 범위 제한 테스트

### 사이드 이펙트 검증

- 이벤트 재처리가 중복 데이터를 만들지 않는지 확인
- DB 캐시가 온체인 원본과 다른 경우 온체인 기준으로 복구되는지 확인
- 컨트랙트 이벤트 필드 변경이 DB 코드에 반영되어 있는지 확인
- R2 파일 저장 정책이 증빙 해시 등록 컨트랙트 흐름과 충돌하지 않는지 확인
- 동기화 실패가 다음 Cron 실행에서 복구 가능한지 확인
- Cloudflare D1 또는 선택한 DB가 Phase 9 API 조회 요구사항을 처리할 수 있는지 확인

### 누적 검증

- Phase 1~7 컨트랙트 테스트 전체 실행
- DB 마이그레이션 및 이벤트 동기화 테스트 실행
- 컨트랙트 이벤트 발생 후 DB 반영까지 통합 검증

### 완료 기준

## Phase 9. Cloudflare API 및 오프체인 데이터 처리 구현

### 목표

프론트엔드가 사용할 Workers/Pages Functions 기반 API와 오프체인 원문 저장, R2 증빙 업로드 흐름을 구현한다.

### 구현

- Cloudflare Workers/Pages Functions에서 실행 가능한 API handler 구현
- 내 DAO 목록 조회 API
- DAO 상세 조회 API
- 제안 상세 원문 저장 API
- canonical JSON 생성 및 `keccak256` 해시 생성 로직
- 제안 취소 사유 저장 API
- 취소 사유 해시 생성 로직
- R2 증빙 파일 업로드 API 또는 업로드 URL 생성 API
- 증빙 파일 메타데이터 저장 API
- 증빙 파일 SHA-256 해시 생성 로직
- 예산 내역 조회 API
- 비구성원 상세 조회 차단 로직
- Cloudflare Secrets 기반 RPC URL, DB, R2 설정 로딩
- 컨트랙트 배포용 private key가 API 런타임에 필요하지 않도록 분리

### 코드 리뷰

- API가 온체인 주소 기준 권한 검증을 수행하는지 확인
- canonical JSON 정렬, 주소 소문자화, wei 문자열화가 문서와 일치하는지 확인
- 원문과 해시가 같은 데이터 기준으로 생성되는지 확인
- 비구성원에게 제안/증빙/예산 상세가 노출되지 않는지 확인
- 증빙 파일 원문이 R2에 저장되고 DB에는 메타데이터만 저장되는지 확인
- R2 object key를 사용자가 임의 조작해 다른 DAO 자료에 접근할 수 없는지 확인
- API 코드가 Workers 런타임에서 지원하지 않는 Node.js API에 의존하지 않는지 확인

- 제안 상세 저장 후 해시가 컨트랙트 입력값과 일치하는지 확인
- 취소 사유 저장 후 해시가 컨트랙트 입력값과 일치하는지 확인
- 증빙 파일 해시가 온체인 등록값과 일치하는지 확인
- 비구성원 접근 차단 확인
- R2 업로드 후 DB 메타데이터와 object key가 일치하는지 확인
- Cloudflare Secrets 누락 시 명확한 오류가 발생하는지 확인

### 테스트

- API 권한 테스트
- DAO 목록/상세/예산 내역 API 테스트
- R2 업로드/메타데이터 저장 테스트
- Workers handler 단위 테스트
- 환경 변수/Secret 검증 테스트

### 사이드 이펙트 검증

- API가 DB 캐시만 믿고 온체인 권한과 어긋나지 않는지 확인
- 원문 수정이 해시 불일치를 만들 경우 감지 가능한지 확인
